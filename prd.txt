# **Product Requirements Document (PRD)**

### **Solar + Survey Equipment Operational Dashboard**

---

## **1. Overview**

**Product Name:** FieldOps Asset Dashboard
**Purpose:** A multi-site operational dashboard that tracks high-value solar and land-survey equipment, automatically surfaces issues (calibration, firmware, damage vs replacement needs), and is field-usable via barcode scanning.

**Users:**
• 1 Admin (global)
• ~50 Field Users (assigned to sites)
• Optional Viewer (read-only, internal stakeholders)

**Platform:**
• Web application (tablet/desktop first)
• Responsive for sunlight readability
• No native app required for MVP

**Branding Colors:**

| Purpose                | Color   |
| ---------------------- | ------- |
| Primary UI             | #001C77 |
| Secondary / Highlights | #00C6C1 |
| Warning                | #FFDA00 |
| Alert / High Risk      | #F75C03 |

---

## **2. High-Level Goals**

This tool must:

1. Replace Excel with a centralized system.
2. Provide actionable insights: “This is what I have. This is what I need. Here are the problems.”
3. Allow quick field scanning of equipment to update records.
4. Track maintenance schedules (calibration + firmware).
5. Distinguish between damage vs simple replacement needs.
6. Present a high-level dashboard only the admin can access.
7. Support ~50 field users with role-based access control.

## **4. Core Functional Requirements**

---

### **4.1 Asset Registry**

Each equipment item should have:

| Field                     | Type        | Notes                       |
| ------------------------- | ----------- | --------------------------- |
| asset_id                  | UUID        | Unique                      |
| asset_number              | string      | barcode id                  |
| part_number               | string      | auto populated              |
| serial_number             | string      | auto populated              |
| equipment_type            | string      | survey / GNSS / solar etc   |
| site                      | foreign key | site assignment             |
| firmware_version          | string      | 128 char max                |
| last_calibration_date     | date        | user editable               |
| calibration_interval_days | integer     | per item                    |
| next_calibration_due      | date        | auto computed               |
| calibration_status        | enum        | OK / DueSoon / Overdue      |
| damage_status             | enum        | OK / Reported / UnderRepair |
| repair_notes              | text        | free form                   |
| cost                      | decimal     | currency                    |
| replacement_cost          | decimal     | currency                    |
| acquired_date             | date        | used for depreciation       |
| created_at                | datetime    | auto                        |
| updated_at                | datetime    | auto                        |

---

### **4.2 Barcode Scanner Integration**

**Requirement:**
Field users should scan an asset’s barcode to:

• Auto-lookup its record
• Populate serial + part info
• Present maintenance status
• Allow quick calibration or damage logs

**Implementation:**
• Use `Html5-QRCode` for web camera scanning
• Fallback text entry for non-camera devices

---

### **4.3 Calibration Logic**

Each item’s calibration schedule is determined by:

```
next_calibration_due = last_calibration_date + calibration_interval_days
```

Status rules:

| Days Until Due | Status   | Color   |
| -------------- | -------- | ------- |
| > 30 days      | OK       | #00C6C1 |
| 21-30 days     | Warning  | #FFDA00 |
| < 21 days      | Due Soon | #F75C03 |
| Past due       | Overdue  | Red     |

---

### **4.4 Firmware Tracking**

Track firmware version and status:

• `latest_firmware_version` (derived)
• `current_firmware_version`
• flag items needing updates
• sortable by firmware risk

---

### **4.5 Damage & Repair Logging**

Field users can log:

• Damage type (drop, abrasion, sensor issue, etc)
• Photo upload (optional)
• Estimated repair cost
• Field remarks

If repair cost > replacement_cost × threshold (configurable), suggest replacement.

---

### **4.6 Depreciation Calculation**

From spreadsheet logic:

```
days_in_service = today - acquired_date
daily_depr_rate = cost / 1095
current_value = max(cost - (daily_depr_rate * days_in_service), 0)
```

Recompute dynamically on dashboard; store value for audit logs.

## **5. Dashboard**

---

### **5.1 Admin Dashboard (Global)**

**Views Must Include:**

#### **A. Risk Triage Table**

• Sites with highest overdue calibration counts
• Firmware out-of-date assets
• Under-repair assets
• Total assets by risk status

#### **B. Value Summary**

• Total asset value vs depreciated value
• Replacement cost estimates
• Repair cost outlays

#### **C. Upcoming Alerts**

• Next 30 days calibration due
• Firmware updates required
• Recently damaged assets

Filterable by:

• Site
• Equipment type
• Vendor
• Risk status

Use brand colors for status highlighting.

---

### **5.2 Field View Dashboard (Per-Site)**

• Only equipment for assigned site
• Maintenance status
• Quick scan → update workflow
• Simple triage list:

* Green: OK
* Yellow: Upcoming
* Orange: High risk
* Red: Past due
